Entities and Entity Relationship Models ( 0.25 h)

An ER Model (or diagram) is a graphical representation of the data structure often used during initial phases of database design.  It can be used for effective communication among stakeholders in a project, and it is often used as the basis from which normalization is performed leading to a final database architecture. 

  •  notean entity can be defined as a person, place, event, or ‘thing’ that exists
  • examples:  person, employee, company, publisher, author, patient, book, motherboard, etc.
  • generally, we wish to be able to store data that describes an instance of the entity; tables are used for this

Relationships:

  • "associations" normally between entities, for example an instance of a customer might be associated with

Identifying Parent/Child relationships between entities is very important when designing databases.

    Different types of attributes exist:

    • an atomic attribute is in its simplest form and cannot be divided; i.e. age
    • a composite attribute can be divided into smaller subparts which represent more basic attributes; i.e. Name could be subdivided into the atomic attributes Last Name and First Name
    • the way the user intends to use the data will determine whether to express an attribute as a composite attribute or to express it in its atomic form; a normalized database always has data stored in atomic form!


    stored versus derived:

    • a stored attribute has its value physically stored in the database
    • a derived attribute has its value determined by the DBMS whenever the attribute is used; i.e. a calculation could be involved that is for example, based on values drawn from 3 columns in a table

    Primary Key (or key attribute):

    • a key attribute (Primary Key) has a unique value for each individual entity; i.e. the social insurance number attribute would be unique for each employee entity in an employees table
    • sometimes several attributes together form a key, meaning that the combination of attribute values is unique for each individual entity; this is called a "composite primary key"

    Primary, Alternate, and Foreign Keys Continued...

    "Candidate Keys" are one or more attributes that could possibly be used to uniquely identify each instance of an entity.  One candidate key in each table is designated as a Primary Key and the others are known as alternate keys.

  • note that primary key column(s) are underlined on an ER diagram

  • once we transform our logical model (ER model) into a physical model, we sometimes may decide to switch our primary key for one of the alternates due to performance and/or storage considerations (this is an advanced topic beyond the scope of this course)

  • Foreign keys are used to relate child record(s) to a parent record (as the key originates in the parent entity, it is considered foreign to the child record).  Foreign keys are sometimes indicated on diagrams using "(FK)" or "(fk)" as shown in the diagram below (also note another way to indicate cardinality is shown):




     

  • When a column or a combination of columns is defined to be a Foreign Key, child records cannot be created in the child table unless the parent record already exists in the parent table (your instructor may wish to expand on this with examples);  this restriction is generally called a "Foreign Key (or Referential Integrity) Constraint".
